Troubleshooting

Häufige Probleme & Lösungen.

Hier findest du Lösungen zu den häufigsten Problemen — von Verbindungsfehlern bis zu Laufwerks-Geistern.

Logs einsehen

Wenn etwas nicht funktioniert, ist der erste Schritt immer ein Blick in die Log-Datei. Logs werden automatisch gespeichert unter: 

%APPDATA%\SSHWinManager\app.log

Debug-Modus aktivieren

Im Debug-Modus werden deutlich detailliertere Informationen protokolliert — hilfreich bei schwer reproduzierbaren Fehlern.

  1. Einstellungen öffnen.
  2. Debug-Modus aktivieren.
  3. Den Schalter Debug in der Hauptoberfläche klicken — es öffnet sich ein Live-Log-Fenster, das alle Ereignisse in Echtzeit anzeigt.
Live-Logs Sobald der Debug-Modus aktiv ist, kannst du über den Debug-Button in der App ein Live-Fenster öffnen, das alle Logs in Echtzeit scrollt — ideal zum Beobachten während du eine Verbindung versuchst.

Laufwerke lassen sich nicht trennen / Geister-Laufwerke

Es kann vorkommen, dass ein Laufwerk nach dem Unmounten im Explorer sichtbar bleibt oder sich gar nicht erst trennen lässt. Das passiert typischerweise, wenn:

  • die SSH-Verbindung unerwartet abgebrochen ist,
  • ein Prozess noch auf das Laufwerk zugreift,
  • Windows den Laufwerksbuchstaben intern noch als belegt führt.

Dafür gibt es in den Einstellungen zwei Hilfsfunktionen:

Fix Ghost Drive

Entfernt Laufwerks-Einträge, die als „Karteileiche" übrig geblieben sind — also Buchstaben, die im Explorer erscheinen, aber zu keiner aktiven SSHFS-Verbindung mehr gehören.

Einstellungen → Fix Ghost Drive

Restart Windows Explorer

Startet den Windows Explorer-Prozess neu. Behebt Fälle, in denen Laufwerke im Explorer hängen bleiben, obwohl SSHFS sie bereits freigegeben hat.

Einstellungen → Restart Windows Explorer

Explorer-Neustart Beim Neustart des Explorers flackert die Taskleiste kurz — das ist normal. Offene Explorer-Fenster werden dabei geschlossen und neu geöffnet.

Authentifizierungs-Probleme

Verbindung schlägt fehl, obwohl Zugangsdaten stimmen? Hier sind die häufigsten Ursachen:

Welcher Auth-Modus ist aktiv?

NEO SSH-Win Manager unterscheidet drei Authentifizierungs-Modi pro Verbindung:

Modus Wann aktiv Verhalten
Passwort Passwort gespeichert, kein „Jedes Mal fragen" Automatischer Login mit gespeichertem Passwort.
SSH-Key SSH-Key hinterlegt, kein „Jedes Mal fragen" Automatischer Login mit hinterlegtem Schlüssel.
Jedes Mal fragen Option aktiviert, Passwort und/oder Key konfiguriert Vor jeder Verbindung erscheint ein Dialog — du wählst, ob Passwort oder SSH-Key verwendet werden soll.
Kein Passwort Kein Passwort, kein Key, kein „Jedes Mal fragen" Die App fragt beim Verbinden automatisch nach dem Passwort.
„Jedes Mal fragen" bedeutet: Methode wählen Diese Option fragt nicht nach dem Passwort — sie lässt dich vor jeder Verbindung auswählen, welche gespeicherte Methode (Passwort oder SSH-Key) genutzt werden soll. Voraussetzung: mindestens eine Methode muss konfiguriert sein.

Checkliste bei Login-Fehlern

  • Stimmen Benutzername und Hostname / IP?
  • Ist der SSH-Port korrekt (Standard: 22)?
  • Ist der SSH-Dienst auf dem Server aktiv?
  • Ist Passwort-Authentifizierung auf dem Server erlaubt (PasswordAuthentication yes in sshd_config)?
  • Bei SSH-Key: Ist der öffentliche Schlüssel in ~/.ssh/authorized_keys auf dem Server eingetragen?
  • Debug-Modus aktivieren und Fehlermeldung im Live-Log prüfen.

Laufwerk lässt sich nicht mounten

Voraussetzungen prüfen

Das Mounten setzt zwei externe Programme voraus. Sind beide installiert?

  • WinFsp — winfsp/winfsp
  • SSHFS-Win — winfsp/sshfs-win

Installationsreihenfolge beachten: zuerst WinFsp, dann SSHFS-Win.

Laufwerksbuchstabe bereits belegt

Wenn der gewünschte Laufwerksbuchstabe bereits von einem anderen Laufwerk genutzt wird, schlägt das Mounten fehl. Entweder einen anderen Buchstaben in den Verbindungseinstellungen wählen, oder die Auto-Erkennung nutzen.

Firewall / Netzwerk

Stelle sicher, dass Port 22 (oder der konfigurierte SSH-Port) vom Client aus erreichbar ist. Ein kurzer Test: 

ssh benutzername@hostname -p 22

Funktioniert das Terminal-Login, liegt das Problem nicht am Netzwerk.

Windows SmartScreen-Warnung

Die App ist nicht code-signiert (Open-Source-Projekt ohne Zertifikat). Windows zeigt beim ersten Start eine SmartScreen-Warnung.

  1. Im Warndialog auf „Weitere Informationen" klicken.
  2. Dann „Trotzdem ausführen" wählen.
Nur aus offizieller Quelle laden Lade die App ausschließlich von den offiziellen GitHub-Releases herunter. SHA256-Prüfsummen sind im Changelog angegeben.
